AI 코딩 도구의 사용은 이제 보편적입니다. [JetBrains 조사 기준 개발자의 90%가 AI 도구를 사용](https://blog.jetbrains.com/research/2026/04/which-ai-coding-tools-do-developers-actually-use-at-work/)하고 있고, [Cursor 3는 에이전트 퍼스트 인터페이스로 전환](https://cursor.com/changelog/3-0)했습니다. 바이브 코딩 도구의 확산으로 [App Store 새 앱 제출 건수가 전년 대비 84% 급증](https://www.theinformation.com/articles/vibe-coding-effect-apples-app-store-saw-84-jump-new-apps-quarter)했다는 보도도 있습니다.

그런데 도구를 쓴다고 생산성이 올라가는 건 아니었습니다. AI가 한 시간에 수천 줄을 생성할 수 있게 되면서, 병목은 "코드를 쓰는 속도"에서 **"코드를 검증하는 속도"**로 옮겨갔습니다. [CodeRabbit의 470개 PR 분석](https://www.businesswire.com/news/home/20251217666881/en/)에 따르면 AI 코드는 인간 코드 대비 결함이 1.7배 더 많고, 일부 분석에서는 보안 취약점이 2.74배 더 많다는 결과도 있습니다.

프롬프트를 잘 쓰는 것만으로는 이 격차를 줄일 수 없습니다. 에이전트가 **처음부터 올바른 방향으로 동작하도록 환경 자체를 설계**해야 합니다. 이것이 하네스 엔지니어링입니다.

## 하네스 엔지니어링이란

[Anthropic 엔지니어링 블로그](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents), [Martin Fowler](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html), [OpenAI](https://openai.com/index/harness-engineering/)가 공통적으로 사용하는 정의를 종합하면:

> **Agent = Model + Harness**

모델은 통제할 수 없습니다. 하지만 하네스는 통제할 수 있습니다. 하네스란 AI 코딩 에이전트가 동작하는 **환경, 규칙, 도구, 워크플로우의 총체**입니다. 프롬프트 엔지니어링이 "무엇을 요청하느냐"의 문제라면, 하네스 엔지니어링은 **"에이전트가 어떤 환경에서 동작하느냐"의 문제**입니다.

Martin Fowler는 이를 이렇게 표현했습니다 — "에이전트에게 '좋은 코드를 작성하라'고 말하는 대신, 좋은 코드가 무엇인지 기계적으로 강제하라."

| | 프롬프트 엔지니어링 | 하네스 엔지니어링 |
|---|---|---|
| **대상** | 단일 요청 | 에이전트의 전체 동작 환경 |
| **지속성** | 일회성 | 프로젝트 수명 동안 누적 |
| **제어 방식** | 자연어로 요청 | 규칙, 도구, Hook으로 강제 |
| **비유** | 운전자에게 "조심해"라고 말하기 | 가드레일과 신호등 설치 |

## 하네스의 5가지 축

하네스를 구성하는 축은 다섯 가지입니다.

### Context — 문맥 공급

에이전트에게 프로젝트의 아키텍처, 컨벤션, 도메인 지식을 구조적으로 전달하는 것입니다. 컨텍스트 없는 에이전트는 프로젝트를 처음 보는 주니어 개발자와 같습니다.

프로젝트에서 CQRS 패턴과 DataSource 라우팅(Command → Main DB, Query → Read Replica)을 사용하고 있었는데, 이 구조를 하네스에 명시하기 전에는 에이전트가 Query Service에서 Command Repository를 호출하는 코드를 종종 생성했습니다. **어떤 레이어에서 어떤 Repository를 사용할 수 있는지를 표 하나로 정리한 것**만으로 이 문제가 사라졌습니다.

핵심은 두 가지였습니다. 전역 규칙 → 프로젝트 규칙 → 도메인별 규칙 순으로 **계층적으로 로딩**하는 것, 그리고 하나의 거대한 문서 대신 역할별로 분리된 **짧은 규칙 파일**들을 사용하는 것. 복잡한 도메인 지식은 별도 문서로 분리하여 필요할 때만 참조하도록 구성하면 토큰 낭비를 줄일 수 있습니다.

### Constraint — 제약 조건

에이전트가 "하지 말아야 할 것"을 명시적으로 정의하는 것입니다.

제약은 반드시 **실제 사고에서 추출해야 합니다**. 이론적으로 "이것도 규칙으로 넣을까?"하고 추가한 것들은 대부분 에이전트가 이미 잘 하고 있는 것들이었습니다. 반면, 실제로 반복된 실수들 — JPA `save()` 호출 후 반환값을 쓰지 않아 `@GeneratedValue` ID가 null이 되는 패턴, `Collectors.toMap()`에서 merge function을 빼먹어 중복 키 런타임 에러가 나는 패턴 — 이런 것들을 규칙으로 넣었을 때 즉각적인 효과가 있었습니다.

같은 사고가 두 번 일어나면, 그건 하네스의 실패입니다.

### Memory — 기억 체계

세션을 넘어 지속되는 학습입니다. 에이전트가 같은 실수를 반복하지 않고, 사용자의 피드백이 축적되도록 하는 구조입니다.

가장 효과적이었던 유형은 **피드백 메모리**입니다. "테스트에서 `mock(Entity.class)` 대신 Fixture 객체를 사용하라"는 피드백을 한 번 기록하자, 이후 모든 세션에서 에이전트가 자동으로 Fixture 패턴을 선택했습니다. 같은 교정을 두 번 할 필요가 없어졌습니다.

한 발 더 나아가, **에이전트가 사용자를 코칭하는 메모리**도 실험 중입니다. AI 활용 습관의 약점(에러 발생 시 근본 원인 분석 없이 "고쳐줘"라고 하는 패턴)을 에이전트가 기억하고 있다가, 적절한 타이밍에 부드럽게 코칭하는 구조입니다.

### Safety — 안전 장치

에이전트의 파괴적 동작을 사전에 차단하는 것입니다. 프로덕션 코드베이스에서 AI 에이전트를 사용할 때 가장 중요한 축입니다.

안전 장치를 만들게 된 계기는 명확합니다 — 에이전트가 `git checkout .`으로 작업 중인 변경사항을 날린 사건이 있었습니다. 그 후 파괴적 명령(force push, hard reset, 파일 삭제 등)을 Hook으로 사전 차단하는 구조를 도입했습니다. 공유 브랜치 보호, 민감 파일 커밋 방지, 빌드 검증 강제도 함께 넣었습니다.

[커스텀 MCP 서버를 만들 때도]({{ site.baseurl }}{% post_url 2026-04-11-custom-mcp-server-for-developer-productivity %}) 같은 원칙을 적용했습니다. DB는 읽기 전용만, API는 멱등성이 보장되는 GET만, 환경은 dev만 허용. 도구를 쥐어주는 것은, 동시에 도구의 경계를 정하는 것이기도 합니다.

### Workflow — 작업 흐름

복잡한 작업을 단계별로 분해하고, 각 단계마다 검증 포인트를 삽입하는 구조입니다.

3개 이상의 파일을 변경하는 구현 작업에서는 반드시 **"계획 → 승인 → 단계별 실행 → 검증"** 사이클을 따르도록 했습니다. 각 단계마다 빌드 체크와 커밋을 수행하기 때문에, 중간에 문제가 생겨도 해당 단계만 롤백하면 됩니다. 에이전트가 10개 파일을 한꺼번에 수정한 후 빌드가 깨지면, 어디서 문제가 생겼는지 추적하는 데 원래 구현보다 더 많은 시간이 듭니다.

Anthropic의 엔지니어링 블로그에서 소개한 [Two-Agent 패턴](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)이 대표적입니다. Initializer 에이전트가 컨텍스트를 수집하고, Coding 에이전트가 실제 구현을 수행하는 분업 구조입니다. [InfoQ에서 보도한 Three-Agent 패턴](https://www.infoq.com/news/2026/04/anthropic-three-agent-harness-ai/)은 여기에 검증 에이전트를 추가한 것입니다.

## 시작한다면

시작은 `CLAUDE.md`(또는 `.cursorrules`) 하나면 됩니다. [HumanLayer의 가이드](https://www.humanlayer.dev/blog/writing-a-good-claude-md)에 따르면 핵심만 간결하게 유지하는 것이 좋습니다. 거기에 MCP 서버 2~3개(GitHub, DB, Search)를 연동하고, 에이전트가 실수할 때마다 규칙을 추가하면 됩니다. 프로젝트에 맞는 조회가 필요하면 [커스텀 MCP 서버를 만드는 것]({{ site.baseurl }}{% post_url 2026-04-11-custom-mcp-server-for-developer-productivity %})도 방법입니다.

## 아직 부족한 것들

하네스 엔지니어링은 아직 초기 단계입니다. 운영하면서 느낀 부족함을 솔직하게 정리합니다.

**하네스 자체의 테스트 방법이 없습니다.** 코드에는 테스트를 작성하지만, 하네스 규칙이 의도대로 작동하는지 검증하는 체계가 없습니다. "JDBC bulk insert 시 audit 컬럼 4개를 반드시 포함하라"는 규칙이 있는데, 새 Entity가 추가될 때 이 규칙이 지켜졌는지는 빌드가 깨져야만 알 수 있습니다.

**효과 측정 지표가 없습니다.** "하네스 적용 후 생산성이 올랐다"는 체감은 있지만, 에이전트의 첫 시도 빌드 성공률, 동일 실수 재발률 같은 정량적 메트릭을 자동 수집하는 체계가 필요합니다.

**메모리가 쌓이면 충돌합니다.** 오래된 규칙과 새 규칙이 충돌하거나, 코드베이스가 바뀌었는데 메모리는 그대로인 경우가 생깁니다. 메모리의 유효성 검증과 자동 정리가 아직 확립되지 않았습니다.

**팀으로 확장하기 어렵습니다.** 프로젝트 규칙은 Git으로 공유할 수 있지만, 개인 워크플로우와 팀 표준의 경계를 어디에 그을 것인가는 열린 문제입니다.

## 돌아보며

하네스를 운영하면서 가장 크게 느낀 건, 에이전트의 실력은 모델 업그레이드가 아니라 하네스의 누적에서 온다는 점입니다. 같은 모델이라도 하네스가 잘 갖춰진 환경에서는 확연히 다른 결과를 냅니다. Context, Constraint, Memory, Safety, Workflow 중 하나라도 빠지면 에이전트는 영리하지만 예측할 수 없는 동료가 됩니다.

아직 부족한 것들이 많습니다. 하지만 설정 파일 하나에서 시작해 에이전트가 실수할 때마다 규칙을 추가하는 것만으로도, 몇 주 후에는 프로젝트에 맞는 하네스가 만들어집니다. AI 도구를 쓰는 것과 AI 도구의 환경을 설계하는 것은 다른 일이고, 그 차이가 결과의 차이를 만들고 있습니다.

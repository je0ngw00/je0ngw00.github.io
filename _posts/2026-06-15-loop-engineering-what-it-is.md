"이제 harness engineering은 한물갔고 loop engineering이 대세다"라는 말을 들었습니다. 정확히 무슨 뜻인지 정리해 두려고 출처를 하나씩 살펴봤는데, 그 전제부터 어긋났습니다. "harness에서 loop로 갈아탔다"는 말은 원문을 따라가 보면 성립하지 않았습니다. 그래서 loop engineering이 실제로 무엇이고 어떤 부품으로 이뤄지는지, Claude Code와 Codex에서는 각각 어떻게 부르는지부터 정리했습니다. 직접 조사하며 든 생각은 맨 끝에 짧게 적겠습니다.

## 프롬프트를 짜지 않고 루프를 짠다

Addy Osmani는 [loop engineering](https://addyosmani.com/blog/loop-engineering)을 "하네스 한 층 위(one floor above the harness)"로 설명합니다. 하네스가 사라지고 루프가 그 자리를 차지한 게 아니라, 하네스 위에 한 단계가 더 얹힌 구조라는 것입니다. 하네스가 에이전트 한 번의 실행 환경(프롬프트, 샌드박스, 도구)을 짠다면, 루프는 그 위에서 타이머로 돌고 보조 에이전트를 띄우며 스스로 다음 작업을 먹여 넣습니다.

이 흐름을 가장 인상적으로 요약한 건 Anthropic에서 Claude Code를 총괄하는 Boris Cherny의 2026년 발언입니다.

> I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops.

같은 방향의 말을 Peter Steinberger도 했습니다 — "이제 코딩 에이전트에 프롬프트를 쓰지 말고, 에이전트에게 프롬프트를 던지는 루프를 설계하라." 지렛대의 받침점이 프롬프트 한 줄에서 루프 한 벌로 옮겨갔다는 이야기입니다.

사실 "행동하고, 결과를 보고, 다시 판단한다"는 반복 자체는 새로운 게 아닙니다. 뿌리를 따지면 reason → act → observe를 교차하는 ReAct 패턴까지 거슬러 올라갑니다. loop engineering은 그 오래된 반복을 코딩 에이전트라는 맥락에서 다시 호명한 이름에 가깝습니다.

여기서 첫 번째로 짚어둘 점이 나옵니다. 새 용어가 뜰 때 가장 먼저 확인할 것은 "교체인지, 한 층 추가인지"입니다. loop engineering은 후자였습니다.

## 루프를 이루는 여섯 가지

그럼 루프는 무엇으로 만들까요. Addy Osmani가 꼽는 부품은 다섯 가지에 기억장치 하나가 더해진 형태입니다. 1년 전이라면 이 부품들을 셸 스크립트로 직접 짜고 관리해야 했지만, 지금은 제품에 기본으로 들어가 있습니다.

- 발견·분류(triage): 정해진 주기에 사람 개입 없이 이슈·로그·작업 보드를 훑어 할 일을 추리고 우선순위를 매깁니다.
- 워크트리: 에이전트마다 격리된 작업 폴더를 줘서, 한 에이전트의 수정이 다른 에이전트의 폴더에 닿지 않게 합니다. 병렬 실행의 충돌을 막는 장치입니다.
- 스킬: 프로젝트 지식을 파일(`SKILL.md`)에 적어두고, 세션이 바뀌어 맥락이 백지로 돌아가도 에이전트가 매번 다시 읽고 시작하게 합니다.
- 플러그인·커넥터: 에이전트를 외부 도구(이슈 트래커, DB, Slack 등)에 연결합니다. 보통 MCP를 통하며, "고치세요"라고 말만 하는 게 아니라 직접 PR을 열고 티켓을 잇습니다.
- 서브에이전트: 코드를 짠 모델은 자기 결과를 후하게 보는 경향이 있어, 만드는 에이전트와 검사하는 에이전트를 일부러 나눕니다.
- 상태 파일: 부품 다섯이 손발이라면 상태 파일은 기억입니다. 대화 맥락 바깥의 마크다운 파일이나 이슈 보드에 무엇을 했고 무엇이 남았는지를 적어, 다음 실행이 멈춘 지점부터 이어집니다. Addy Osmani의 표현으로는 "에이전트는 잊어도, 저장소는 잊지 않는다"입니다.

## 같은 부품, 다른 이름 — Claude Code와 Codex

흥미로운 건 이 여섯 부품이 Claude Code와 Codex 양쪽에 모두 들어가 있고, 갈리는 건 기능이 아니라 이름과 설정 위치라는 점입니다.

| 부품 | Claude Code | Codex |
|---|---|---|
| 발견·분류 / 자동 실행 | `/loop`, `/goal`, cron·hooks | Automations 탭, Triage inbox, `/goal` |
| 워크트리 | `claude --worktree`, 서브에이전트 `isolation: worktree` | 스레드별 워크트리 내장 |
| 스킬 | `SKILL.md` | `SKILL.md`(동일 형식), `$이름` 호출 |
| 커넥터 | MCP 서버 + 플러그인 | MCP + 플러그인 |
| 서브에이전트 | `.claude/agents/` | `.codex/agents/` |
| 상태·기억 | 마크다운(AGENTS.md) 또는 Linear | 마크다운 또는 Linear |

어느 쪽을 쓰든 "루프를 짠다"는 사고방식은 그대로 옮겨갑니다. 부품의 유무가 아니라 부르는 이름만 다를 뿐입니다.

## Ralph loop와 무엇이 다른가

루프 자동화가 새로운 거라면, 기존에 있던 루프 기법과는 뭐가 다른지가 자연스러운 질문입니다. 가장 유명한 선행 기법은 Geoffrey Huntley가 정리한 [Ralph loop](https://ghuntley.com/ralph)입니다. 한 줄짜리 무한 반복입니다.

```bash
while :; do cat PROMPT.md | claude-code ; done
```

처음에는 "Ralph는 검증도 재시도도 없는 무식한 반복이고, loop engineering이 거기에 기능을 더한 것"이라고 정리했는데, 출처를 더 보니 틀린 생각이었습니다. Ralph도 검증·재시도·상태관리가 다 가능했습니다. 다만 그것이 루프 자체의 기능이 아니라, 사람이 프롬프트와 파일에 직접 설계해 넣는 것이었습니다. 같은 프롬프트를 반복해도 에이전트는 매번 `fix_plan.md` 같은 상태 파일을 읽고 다음에 할 일을 스스로 판단합니다. 그래서 텍스트는 고정이어도 행동은 매 회 달라집니다.

Steve Kinney가 이 구조를 [정확하게 짚습니다](https://stevekinney.com/writing/the-ralph-loop).

> 프롬프트가 프로그램이고, bash 루프가 런타임이고, 파일시스템이 상태저장소이고, LLM이 실행 엔진이다.

프로그램 코드는 고정이고 변수만 바뀌는 것과 같습니다. 그러니 차이는 기능의 유무가 아니라, 그 하네스를 사람이 손으로 짜느냐 제품이 부품으로 주느냐입니다. loop engineering은 사람이 손으로 짜던 검증·재시도·상태관리를 제품이 기본 부품으로 흡수한 것이지, 없던 능력이 새로 생긴 게 아닙니다.

유일하게 "구조적으로 더해졌다"고 부를 만한 것은 Claude Code `/goal`의 [독립 검수 모델](https://code.claude.com/docs/en/goal)입니다. 작업을 수행하는 모델과 별개의 모델(기본값은 Haiku)이 완료 여부를 판정하는 구조입니다. Ralph가 자기 작업을 자기가 판정하던 것과는 다릅니다. 다만 이 검수자도 완전하지는 않습니다. 대화 기록만 보고 판정할 뿐 파일이나 테스트를 직접 읽지는 못하기 때문에, "에이전트가 했다고 말한 것"은 검수하지만 "실제로 그렇게 됐는지"까지는 보지 못합니다.

## 냉정하게

루프를 돌린다고 사람이 일에서 빠지는 건 아닙니다. 오히려 루프가 좋아질수록 사람이 붙잡아야 할 부분이 선명해집니다. Addy Osmani도 "당신의 일은 작동을 직접 확인한 코드를 내보내는 것"이라며 검증은 여전히 사람 몫이라고 못 박습니다.

비용도 현실적인 제약입니다. 5분짜리 루프가 매 실행마다 작업자와 검수자를 띄우면 토큰을 빠르게 소진합니다. Claude Code의 쿼리 루프는 재시도할 때마다 전체 대화 기록을 다시 보내기 때문에, 한 프롬프트가 수십만 토큰을 쓰기도 합니다. 그래서 발견·분류는 싸게 돌리고, 서브에이전트는 정말 처리할 일이 있을 때만 띄우는 설계가 권장됩니다. 마침 Anthropic은 구독 플랜에서 사람이 개입하는 사용과 자율로 도는 사용을 분리해 다루기 시작했는데, 루프를 돌릴 때 비용을 의식하라는 신호로 읽힙니다.

## 내 생각

조사를 마치고 든 생각은, loop engineering이 완전히 새로운 패러다임이라기보다 손으로 짜던 것을 제품이 부품으로 흡수한 "실체 있는 정리"에 가깝다는 것입니다. 실무 흐름 자체는 진짜이고 남습니다 — 루프로 에이전트를 굴리는 방식은 계속 쓰일 겁니다. 다만 스케줄링 층을 들여다보면 결국 cron이고, 부품 하나하나는 이전부터 있던 것들이라, "새 능력"이라기보다 "손맛이 제품 기능으로 옮겨간 것"으로 보였습니다.

이 결론이 회의론자의 편향일 수도 있어서, 결론을 숨기고 입장을 갈라 여러 AI에게 따로 물어봤습니다. 한쪽에는 "버즈워드라고 의심해보라", 다른 쪽에는 "진짜 새로운 점을 끝까지 찾아보라"는 식으로요. 대체로 같은 쪽으로 모였습니다.

수치를 볼 때는 출처를 한 번 더 봤습니다. "Claude Code 코드의 80~90%를 AI가 작성한다"는 인상적인 숫자는 Anthropic의 자기보고이고, "엔지니어당 일일 merge PR이 67% 늘었다"는 것도 같은 출처입니다. 그런데 코드 작성 비중이 높다는 것과 생산성이 그만큼 올랐다는 것은 다른 이야기입니다. 오히려 독립적으로 측정한 [METR의 실험](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)에서는 숙련 개발자가 초기 2025년 도구를 쓸 때 19% 느려졌다는 결과가 있었습니다 — 도구 세대가 빠르게 바뀐다는 단서는 붙지만, 벤더 자기보고와 독립 검증 사이의 간극을 보여주는 수치입니다.

한 발 물러서 보면 prompt engineering, context engineering, harness engineering, loop engineering으로 이어지는 이름의 계보가 보입니다. 정작 이 순서조차 글쓴이마다 조금씩 다르게 그립니다. 개념(실무)은 남고 이름은 대략 1~2년마다 갈아탑니다. 그래서 저는 loop engineering이라는 이름에 너무 무게를 두기보다, 그 안의 부품들을 내 작업에 하나씩 끼워보는 쪽이 더 남는 장사라고 생각합니다. 스킬 한 장과 진행 파일 하나부터 시작해도 충분합니다.
